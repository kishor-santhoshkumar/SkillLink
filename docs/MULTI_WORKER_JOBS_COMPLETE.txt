HOW IT WORKS:
================================================================================

WORKER PERSPECTIVE:
1. Browse open jobs
2. Apply to job
3. Get notification when accepted
4. Job remains visible to other workers (still open)
5. Job disappears only when company closes it

COMPANY PERSPECTIVE:
1. Post job
2. View applicants as they apply
3. Accept multiple workers (job stays open)
4. See count: "X worker(s) accepted • Y pending"
5. View profile of any applicant
6. Close job when enough workers hired
7. Reopen job if need more workers

================================================================================
JOB STATUS FLOW:
================================================================================

OLD SYSTEM:
open → assigned (after 1st worker accepted) → completed

NEW SYSTEM:
open → (accept worker 1) → still open
     → (accept worker 2) → still open
     → (accept worker 3) → still open
     → (company clicks "Close Job") → closed
     → (company clicks "Reopen Job") → open again

================================================================================
API ENDPOINTS ADDED:
================================================================================

PATCH /jobs/{job_id}/close
- Close a job opening
- Company only
- Job status: open → closed
- Workers can no longer apply

PATCH /jobs/{job_id}/reopen
- Reopen a closed job
- Company only
- Job status: closed → open
- Workers can apply again

PATCH /jobs/applications/{id}/accept (UPDATED)
- Accept worker application
- Job remains OPEN (not changed to "assigned")
- Multiple workers can be accepted
- Each worker gets notification

DATABASE SCHEMA:
================================================================================

job_requests:
- status: "open" or "closed" (no more "assigned")
- assigned_worker_id: Legacy field (not used)
- Multiple workers tracked via job_applications table

job_applications:
- Multiple applications per job
- Each can be "pending", "accepted", or "rejected"
- Tracks all workers for a job
